### Breaking changes

**Sí — comportamiento cambia en estos flujos:**

1. Usuarios con multi-tenant membership: deben tener `active_tenant_id` en JWT.
   Sesiones existentes se invalidan post-merge (force-global-signout). User
   re-loguea, hook custom_access_token genera nuevo JWT con claim, todo
   funciona transparente.

2. RLS v2 RESTRICTIVE coexiste con v1 PERMISSIVE durante 24h validation. En
   v0.2.6 (próximo sprint) se quema v1 — eso SÍ será breaking si hay usuarios
   pre-v0.2.5 sin haber re-logueado.

3. Email templates ahora vienen de Resend con dominio `mail.impluxa.com` (no
   del default Supabase SMTP). Aceptados por DKIM/SPF verificados en sa-east-1.

### Rollback plan

Si algo se rompe en prod post-merge:

1. **Inmediato:** setear `APPROVAL_GATE_ENABLED=0` en Vercel env vars → deshabilita
   la enforcement application-layer mientras se debuggea. DB sigue protegida por
   v2 RESTRICTIVE (fail-closed by design).

2. **Si RLS v2 niega acceso masivo:** UPDATE `user_session_state SET active_tenant_id = NULL WHERE ...`
   via service-role + force user re-login → repobla con default tenant via
   backfill query original.

3. **Si hook crashea token issuance:** Supabase Dashboard → Auth → Hooks →
   disable Custom Access Token Hook. Cae a JWT sin claim → v2 RESTRICTIVE niega
   pero v1 PERMISSIVE deja pasar (degradación temporal a comportamiento pre-v0.2.5).

4. **Nuclear:** revert merge commit + tag `v0.2.4` deploy → rollback completo.
