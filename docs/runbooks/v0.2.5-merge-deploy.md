## Resumen ejecutivo

v0.2.5 endurece auth/RLS antes de onboardear segundo tenant. 5 amenazas mitigadas, 9 FR cerrados (FR-AUTH-1 a FR-AUTH-9). Diseño en `docs/adrs/0005-auth-re-architecture.md`. Migración requiere force-global-signout de sesiones existentes (~30s downtime percibido al re-login).

**Riesgo principal:** RLS v2 RESTRICTIVE niega acceso por falta de claim `active_tenant_id`. Mitigación: v1 PERMISSIVE coexiste 24h post-merge (fallback fail-safe). v0.2.6 quemará v1.

### Step 2 — Smoketest login real

Hacé esto con tu email real (jota@impluxa.com o el que tengas en Hakuna):

2. Ingresar email → Submit

### Step 3 — Verificar claim active_tenant_id en JWT

1. En DevTools → Application → Cookies → copiar valor del access token
2. Pegarlo en https://jwt.io
3. En el payload debe haber `"active_tenant_id": "<uuid>"`. Si NO está → hook custom_access_token no se enganchó. NO mergear.

### Step 4 — Verificar RLS v2 filtra correcto

Opcional pero recomendado:

1. Si tenés membership en 2+ tenants en la DB de preview: usar el switcher `/admin/tenants` (UI) → cambiar active_tenant → reload `/api/audit` → ver que `tenant_id` cambia en los eventos retornados.

## Rollback paths

### A. Build verde pero algo se rompe en runtime

Setear `APPROVAL_GATE_ENABLED=0` en Vercel → deshabilita enforcement application-layer. DB sigue protegida por RLS v2 RESTRICTIVE.

### B. RLS v2 niega acceso masivo (usuarios sin claim)

Bajo este escenario hay usuarios con `active_tenant_id` no seteado en JWT. Causa probable: hook custom_access_token no se enganchó o falló al issuance.

Query SQL via Supabase Dashboard SQL editor:

```sql
UPDATE user_session_state
SET active_tenant_id = (
  SELECT tenant_id FROM tenant_users WHERE user_id = user_session_state.user_id LIMIT 1
)
WHERE active_tenant_id IS NULL;
```

Después force-global-signout otra vez → re-login → claim repobla.

### C. Hook custom_access_token crashea token issuance

**Síntoma:** todos los logins fallan con 500.

1. Supabase Dashboard → Auth → Hooks → disable Custom Access Token Hook
2. JWT vuelve a no llevar claim → RLS v2 RESTRICTIVE niega, pero v1 PERMISSIVE deja pasar (temporal)
3. Lord Claudia investiga + fix en branch nuevo + nuevo PR

### D. Nuclear rollback

```bash
git checkout main
git revert <merge-commit-hash> -m 1
git push origin main
git tag -d v0.2.5
git push origin :v0.2.5
```

Esto retorna prod a v0.2.4 estado. Vercel auto-deploy. ETA ~5 min.
