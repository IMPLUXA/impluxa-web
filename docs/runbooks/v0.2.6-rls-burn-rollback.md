## Use this runbook when

ALL of the following hold:

1. v0.2.6 burn migration `2026XXXX_v026_burn_v1.sql` was applied to Hakuna prod.
2. Post-burn monitoring (1h intensive window) detects:
   - Spike in 5xx errors on tenant-data routes (`/api/sites`, `/api/leads`, `/api/audit`, etc.), OR
   - Magic link login succeeds but tenant data returns empty / 403, OR
   - `claim_missing` audit events spike when token-mint volume is normal, OR
   - Hakuna user reports "no veo mis datos" within minutes of cutover.
3. APP-layer kill switch (`APPROVAL_GATE_ENABLED=0` in Vercel per ADR-0005) does NOT recover the read path.

If only #1 + (suspicious symptom but not confirmed user impact) → STOP, escalate to consejo (Backend Architect + Security Engineer + Senior PM convene), do NOT roll back yet. False rollbacks during a real incident waste recovery time.

## Pre-flight (before rollback execution)

1. **Snapshot current prod state** for forensic record:
   ```sql
   -- Run via Supabase SQL editor OR CLI with service role
   SELECT row_to_json(p) FROM pg_policies p
   WHERE tablename IN ('sites','leads_tenant','subscriptions','activity_log');
   ```
   Save output to `D:\segundo-cerebro\wiki\meta\hakuna-policies-incident-<TIMESTAMP>.json`.
2. **Capture Vercel runtime logs** for the failing routes (last 30 min). Save to incident dir.
3. **Capture audit_log entries** for the window:
   ```sql
   SELECT * FROM audit_log
   WHERE created_at > <burn_apply_ts>
   ORDER BY created_at DESC
   LIMIT 200;
   ```
4. **Get Rey OK explicit** (gravedad #21.a, prod Hakuna live). Rollback is a write to prod DB; even though it restores known-good state, it is NOT auto-promoted by Lord Claudia. Telegram + chat ASK + Rey "OK procedo".

## Rollback procedure (option B per OQ-1 LOCKED verdict)

### Step 1 — Verify rollback file exists

```bash
ls -la D:/impluxa-web/supabase/migrations/_rollback_snapshots/v0.2.6_v1_policies_FROZEN.sql
```

The file MUST be present. It was captured at burn-migration-write-time via `pg_dump --schema-only --table=public.{sites,leads_tenant,subscriptions,activity_log}` and frozen into the repo. If file is MISSING → STOP, escalate to consejo (this is an architecture violation; rollback cannot proceed safely without snapshot).

### Step 2 — Dry-run on Supabase preview branch

DO NOT apply directly to prod first. Create or reuse a Supabase preview branch:

```bash
# Via Supabase MCP
# 1. Create or reuse branch with current migration state matching prod
# 2. Apply rollback file
# 3. Verify v1 policies restored via pg_policies query
```

If dry-run fails (syntax error, FK conflict, etc.) → STOP, fix snapshot file, repeat.

### Step 3 — Apply to prod (Rey-gated)

```bash
KING_SIGNED=true SUPABASE_DB_URL=<prod-url> \
  supabase db push --include-files supabase/migrations/_rollback_snapshots/v0.2.6_v1_policies_FROZEN.sql --linked
```

Or via Supabase MCP `apply_migration` with explicit migration body (recommended — same auth as Lord Claudia daily ops).

### Step 4 — Verify v1 policies restored

```sql
-- Should now show v1 + v2 policies coexisting (back to v0.2.5 state)
SELECT tablename, policyname, cmd FROM pg_policies
WHERE tablename IN ('sites','leads_tenant','subscriptions','activity_log')
ORDER BY tablename, policyname;
```

Expected: each table has BOTH `*_v1` and `*_v2` policies. Total ≥ 8 rows.

### Step 5 — Verify Hakuna app reads recover

1. Rey re-loguea (logout + magic link) to refresh session.
2. Loads dashboard, checks tenant data visible.
3. Loads audit log, checks events filtered correctly.
4. If app recovered → rollback successful.

### Step 6 — Incident closeout

1. Telegram al Rey: rollback applied + verify result + impact summary (downtime min, # users affected).
2. Update `D:\segundo-cerebro\wiki\meta\session-boot.md` autonomous_decisions_log con entry: rollback applied + reason + new state.
3. Open incident postmortem document `D:\segundo-cerebro\wiki\incidents\v0.2.6-burn-rollback-<DATE>.md` with timeline + root cause hypothesis.
4. **Block any further v0.2.6 burn attempt** until root cause is identified + addressed. Tag GitHub issue with `v0.2.6-burn-blocked`.

## Recovery target (TTR)

- Detection → Rey OK ASK: <5 min
- Rey OK → migration applied: <2 min
- Application reads recover: <1 min after migration commit
- **Total target: <10 min from detection to recovery.**

This compares to git-revert approach (Option A, rejected per OQ-1) which would require new migration commit + push + Supabase project sync = ~25-40 min depending on Supabase deploy lag. Option B's pre-frozen rollback file beats it by >2x.

## Anti-patterns

- **Apply rollback without snapshot file present.** Snapshot is the contract; without it, you are guessing at v1 policy DDL. Better to escalate.
- **Skip Rey OK on rollback because "we already have OK on the burn".** No — those are two distinct prod writes. Each requires its own ASK per regla #21.a.
- **Rollback without capturing forensic state first.** Loses postmortem evidence; you cannot re-investigate root cause once policies are restored.
- **Assume APP-layer kill switch is enough.** `APPROVAL_GATE_ENABLED=0` only opens app-layer enforcement. If RLS v2 has a bug that filters data wrong, the kill switch does NOT help; only DB-layer rollback does.
- **Apply rollback to wrong DB** (preview vs. prod). Always verify connection string before `KING_SIGNED=true`. Prod URL is in `lord-claude.credentials` `SUPABASE_DB_URL_HAKUNA`.
