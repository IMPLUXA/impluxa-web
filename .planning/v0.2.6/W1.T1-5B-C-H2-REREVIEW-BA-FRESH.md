## Q1 — Empirical reproduction of the corruption path

### Step 1: Can a tenant-claim action originate without `jwt_jti`?

Yes. Path:

- `guard.ts:171-191` (`requireActiveTenantOrRedirect`) and `:205-241` (`requireActiveTenantOrResponse`) both invoke `emitTenantClaimAudit(user, result.reason, session?.access_token)` at L187 and L231.
- `session?.access_token` is `string | undefined`. If `getSession()` returns no session (race window: `getUser()` succeeded but cookie expired between the two calls; or any future change replacing `getSession` with a path that returns `{session: null}`), `access_token` is `undefined`.
- `extractJti(accessToken)` at `guard.ts:104-115` returns `undefined` when `accessToken` is undefined (L105), when `decodeJwt` throws (L111 catch), or when `payload.jti` is missing/non-string/empty (L108 check).
- `emitTenantClaimAudit` (L127-152) then calls `writeAuditEvent({ action: reason, jwt_jti: jti, actor_user_id: user.id })` with `jwt_jti: undefined`.

Reachability: **YES.** The undefined-jti branch is reachable any time `getSession()` returns no session OR the access token lacks a JTI claim OR the token is malformed. The token-came-from-`getUser()` invariant prevents most cases but not all (race + future-SDK + JTI-not-issued by gotrue).

### Step 2: What does `audit.ts:86-98` do with that payload?

`audit.ts:86-98`:

```ts
if (
  TENANT_CLAIM_ACTIONS.has(event.action) &&
  (!event.jwt_jti || event.jwt_jti.length === 0)
) {
  console.warn(JSON.stringify({ level: "warn", event: "audit_dedup_bypass_missing_jti", ... }));
}
```

This is a `console.warn` **only**. It does NOT `return`, does NOT `throw`, does NOT short-circuit. Execution falls through to L100-104:

```ts
const supabase = getSupabaseServiceClient();
const { error } = await supabase.rpc("append_audit", { p_event: event });
if (error) { throw new Error(...) }
```

The RPC is called with `p_event.jwt_jti = undefined` (so JSON-serializes to either absent or null).

### Step 3: What does the SQL dedup gate see at line 143?

`migrations/20260518_v026_001_audit_dedup.sql:121-152`:

- L136: `v_jti := nullif(p_event ->> 'jwt_jti', '')::uuid;` — undefined/missing/empty becomes NULL via `nullif`.
- L137-139: malformed UUID is also coerced to NULL via the `exception when invalid_text_representation` block.
- L143: `if v_jti is not null and v_action in ('claim_missing', 'active_tenant_null') then`

When `v_jti IS NULL`, **the entire `if` block (L143-152) is skipped.** No insert into `audit_dedup`, no `return null` short-circuit. Execution falls through to L154-179: `insert into public.audit_log (...) returning id into v_audit_id;`.

**Result: every retry writes a fresh `audit_log` row.** No dedup. Confirmed by static reading of SQL.

### Step 4: Does FR-RLS-BURN-2 readiness flip on this corruption?

`scripts/observe-rls-burn-readiness.ts:189-194` queries `audit_log` for `action=eq.claim_missing` with `count=exact`. Returns raw row count.

`scripts/observe-rls-burn-readiness.ts:249-254`:

```ts
if (c.claimMissing > 0) {
  reasons.push(
    `claim_missing > 0 — hook fail-closed fired. Triage before burn.`,
  );
  verdict = "NO-GO";
}
```

**Threshold is literally `> 0`.** ONE single retry storm event with missing JTI writes N rows into `audit_log`, but even ONE row already triggers NO-GO.

**Corruption confirmed**: a retry storm doesn't have to inflate the count from 0 to 50 to corrupt the gate. The threshold is so strict (>0) that even a single un-deduplicated event blocks the burn. But conversely, the corruption isn't ABOUT the count — it's about whether **the same underlying condition produces the same gate result regardless of retry count**. With dedup gate: one event = one row = 1 > 0 = NO-GO. Without dedup gate: one event with no JTI = N rows = N > 0 = NO-GO. **Same verdict.**

### Verdict on Q1

The corruption path EXISTS mechanically (no dedup when JTI null), but the FR-RLS-BURN-2 gate is `claim_missing > 0` — a binary threshold. The count inflation does NOT change the gate outcome. The gate triggers NO-GO from row #1 onward. A retry storm makes the count noisier but does not flip the verdict.

**Cold pass's empirical claim ("50 events go NO-GO even though underlying is 1 event") is TRUE on the count, but MISLEADING on the gate.** The gate already would have gone NO-GO at count=1. The noise is operational (triage harder) but not signal-corrupting for the burn flip itself.

**HOWEVER**, there's a SECOND-ORDER concern that neither prior pass addressed: per SPEC.md FR-RLS-BURN-2 acceptance criterion explicitly says "zero `claim_missing` audit events **from real users (not synthetic)**". This implies operators triage and discount synthetic/non-real events when deciding the Rey sign-off. Triage cost scales linearly with row count. A retry storm of 500 rows from the same actor would force operators to deduplicate by hand. That IS a real cost, but it's an operational triage cost — NOT a flip-the-readiness-signal corruption.

Noise complicates the human's job but doesn't flip the verdict on its own. Real impact: triage time + ADR-0010 reader confusion ("dedup gate doesn't always dedup").
