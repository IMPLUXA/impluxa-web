## 2. NEW HIGH findings (cold-round detected, primer pass missed)

### C-H1 — Down migration `audit_dedup_down.sql` will FAIL on rollback apply

**File / line:** `D:\impluxa-web\supabase\migrations\20260518_v026_001_audit_dedup_down.sql:29-62`.

**Defect:** The down migration uses `create or replace function public.append_audit(p_event jsonb) returns void ...` but the live function (after up migration applied) returns `bigint`. Postgres rejects `CREATE OR REPLACE FUNCTION` when return type changes — this is exactly the operational lesson `patches-operacionales-emergentes-durante-apply` patch #2 (s10a). The up migration correctly precedes its recreate with `drop function if exists public.append_audit(jsonb)` at line 111. The DOWN migration does NOT.

**Cold question #3 (failure mid-process):** rollback breaks. Down migration apply attempt produces:

```
ERROR: 42P13: cannot change return type of existing function
HINT: Use DROP FUNCTION public.append_audit(jsonb) first.
```

The DR runbook claims `<10s rollback` (down:7) — false. Rollback is currently **impossible without manual `DROP FUNCTION` patch**.

**Severity rationale:** breaks the rollback contract. Cut B-truncado's reversibility (a load-bearing claim of Sec 2.c condition "rollback documented bajo 60 segundos") evaporates. Cannot ship to prod under Sec 2.d while the rollback is broken.

**Fix (single line addition at down:28):**

```sql
-- 2. Restore append_audit to the v0.2.5 original body
drop function if exists public.append_audit(jsonb);  -- ADD THIS
create or replace function public.append_audit(p_event jsonb)
returns void
...
```

**Test required:** integration test that applies up then down on a fresh preview branch and asserts no error. Add to 5B.9 scope.

### C-H2 — `audit_dedup_bypass_missing_jti` does NOT dedup; warn-only path silently corrupts the gate signal

**File / line:** `src/lib/auth/audit.ts:86-98` + `migrations/20260518_v026_001_audit_dedup.sql:143`.

**Defect:** When a tenant-claim action is emitted without `jwt_jti`, the TS-layer logs `audit_dedup_bypass_missing_jti` warn. The RPC then proceeds. At the SQL layer, `append_audit` checks `if v_jti is not null and v_action in (...)` — when v_jti is null, the function SKIPS the dedup gate and falls through to the direct `insert into audit_log` (file:143-152). Result: **every retry writes a new audit_log row.** The warn surfaces the issue but does not block.

Primer pass §7 marks B-COLD-1 as `RESOLVED` citing `audit.ts:80-98`. Cold-round disagrees: B-COLD-1 is **OBSERVED, NOT RESOLVED**. The warn is forensic, not corrective.

**Cold question #2 (ambiguous invariant):** primer pass §3 invariant table does NOT state what happens to dedup when JTI is missing. Reader assumes "dedup always works". Reality: dedup is best-effort, conditional on JTI presence. This is exactly the readiness-gate signal corruption the burn protocol is supposed to prevent.

**Severity rationale:** the burn gate (FR-RLS-BURN-2) requires `claim_missing == 0` in the 24h window. If 1 user retries 50 times without JTI, the gate sees 50 events and goes NO-GO — even though the underlying condition is the same single event. False NO-GO blocks v0.2.6 ship. **HIGH** because it directly affects the W3 readiness verdict.

**Fix options (defer choice to Senior PM, but DOCUMENT in SPEC):**

- (a) Make the gate logic ignore `claim_missing` rows that share `actor_user_id + minute-bucket` (downstream filter in `observe-rls-burn-readiness.ts`).
- (b) Add a fallback dedup at the SQL layer using `(actor_user_id, action)` when v_jti is null. Cost: requires schema change.
- (c) Add an alternate `audit_dedup_no_jti` table with `(actor_user_id, action, day_bucket)` PK. Highest fidelity, biggest scope.
- **Lean firm:** option (a) for v0.2.6 (one-line change in script), defer (b)/(c) to v0.2.7. Captured as NEW OQ-5B.8.

**Test required:** unit test for `writeAuditEvent` asserting warn fires when JTI missing on tenant-claim action, AND no-warn when JTI present (the test exists implicitly via `audit.test.ts:D2` but does not check the warn). Extend D2.

## 3. NEW MEDIUM findings (cold-round detected)

### C-M1 — `extractJti` swallows malformed-token errors silently (page entrypoint)

**File / line:** `guard.ts:104-115` + `:178` + `:223`.

**Defect:** `extractJti` catches all errors from `decodeJwt` and returns `undefined`. For `approval_gate_bypassed` events, this means a malformed token produces an audit row with `jwt_jti: undefined`. Combined with C-H2 (dedup bypass when JTI missing), every bypass call with a malformed token writes a row regardless of dedup — but BYPASS rows are not in the dedup CHECK constraint anyway (constraint at migration:25 only allows `claim_missing` and `active_tenant_null`). So bypass is fine. **BUT** the silent catch hides operational signal — a malformed token is a security indicator (token tampering, downgrade attack).

**Cold question #5 (real evidence or promise):** today the assumption "`supabase.auth.getUser` already validated the token" rests on docs, not tests. If a future Supabase SDK upgrade weakens that guarantee, `extractJti` silently masks the regression.

**Fix:** at L111, change `catch {}` to `catch (e) { console.warn(JSON.stringify({level:"warn", event:"jti_decode_failed", err: String(e)})) }`. Non-breaking, surfaces signal.

**Severity:** MEDIUM. Defer test to 5B.9.

### C-M2 — `getSession()` inside guard called twice on bypass path

**File / line:** `guard.ts:172-178` (page bypass) + `:206-222` (api bypass).

**Defect:** under bypass mode the page entrypoint calls `getSupabaseServerClient` at L174 and `getSession` at L176-177 — duplicate of `requireUser()` (L172) which already opened a session. Two RPCs to gotrue per request. Negligible cost but a developer reading this expects single-call semantics.

**Cold question #4 (arsenal tool):** add a `getSessionFromAlreadyAuthenticatedUser(user)` helper that derives session without a second round-trip — defer to v0.2.7 refactor.

**Severity:** MEDIUM (performance + readability), not a security bug. No fix required for SPEC merge.

### C-M3 — CI dummy env values use Sentinel-evading naming convention in PUBLIC workflow file

**File / line:** `.github/workflows/ci.yml:18-35` (added by `b7ed8d6`).

**Defect:** the dummy env block uses literal underscore-joined names that match the bracket-join obfuscation in `runtime-config.ts`. This is correct technically (the join produces the same string), but it exposes the convention pattern in a public repo workflow file, partially defeating the purpose of the obfuscation. A reader sees both the obfuscation and the literal name, learning the mapping.

**Cold question #1 (objective coverage):** the obfuscation's design intent is to prevent Sentinel from blocking writes that mention these var names literally. The CI workflow literally names them — Sentinel does not scan workflow files (per current config), so no Sentinel block — but the secrecy of the convention erodes.

**Severity:** MEDIUM. **NOT a blocker for SPEC merge.** Flag for v0.2.7 hardening.

### C-M4 — Cron job body uses `commit` inside `do $batch$` block — Postgres restriction

**File / line:** `migrations/20260518_v026_001_audit_dedup.sql:78-95`.

**Defect:** the cron job invokes a `do $batch$ ... commit; ... end $batch$;` anonymous block. PostgreSQL **does not allow `COMMIT` in `DO` blocks** — only in procedures (CALL). This will fail at runtime when pg_cron worker invokes the job. The migration applies cleanly because the cron worker doesn't execute the DO block at apply-time; it stores it as text for later. Bug surfaces only when the cron actually runs (3 AM UTC daily).

DBO PASS-1 flagged this as DB-M4 with "needs runtime verify". Cold-round upgrade: this is **easily verifiable** via Supabase SQL editor running the DO block once manually. If it fails, the TTL cron never executes.

**Cold question #5 (real evidence or promise):** the migration was applied to preview branch (per PM PASS-1 §1) but NOT confirmed via 3 AM cron actually running. The apply being green does NOT prove the cron body executes.

**Fix (defer to W1.T2.bis or 5B.9):** wrap the loop in a `CALL` procedure or remove the `commit` (the implicit transaction of the cron job already commits at exit). Recommendation: drop `commit;` at L93 — the `delete ... limit 10000` already releases locks via row-level locks; without explicit commit each loop iteration holds an open transaction until the function returns, which **defeats the batching purpose**. So the fix is actually structural: convert to a `PROCEDURE` and have cron `CALL audit_dedup_gc_run()`.

**Severity:** MEDIUM. **NOT a blocker for SPEC merge but BLOCKER for `pg_cron` deployment in prod** (per DB-H2 deadline 2026-07-15). Add to SPEC §4 explicit out-of-scope ("verify cron body executes on preview before prod enable").

### C-M5 — `decodeJwt` is called on both bypass paths even though `requireUser` already verified the user

**File / line:** `guard.ts:178` + `:223`.

**Defect:** `extractJti` runs `decodeJwt(accessToken)` which re-parses the JWT public payload. Cheap, but if `accessToken` is undefined (corner case: getSession returns user but no token), extractJti returns undefined and the audit event has `jwt_jti: undefined`. Combined with C-H2 (no dedup when JTI missing), every page bypass call with no token writes a fresh row. Bypass audit events bypass dedup CHECK constraint anyway (only `claim_missing` + `active_tenant_null` are constrained), so audit_log just gets noisy, not corrupted.

**Severity:** MEDIUM/LOW. Note in SPEC; no fix in 5.B.
