## 6. Migration safety re-verify (C-H1)

**Concern**: does the C-H1 fix in `audit_dedup_down.sql:28-35` actually prevent the 42P13 error?

**Evidence read from file**:

- Line 36: `drop function if exists public.append_audit(jsonb);`
- Line 38-71: `create or replace function public.append_audit(p_event jsonb) returns void ...`

**Analysis**: PG semantics for `42P13` apply when `CREATE OR REPLACE FUNCTION` attempts to change return type of an existing function (here: `bigint` from the up migration → `void` in the down). The `drop function if exists` at line 36 removes the existing `bigint` signature first. The subsequent `create or replace function` then creates a fresh function with `returns void` — no return-type conflict.

**Sub-concern**: argument-list mismatch could also block. The up migration creates `append_audit(p_event jsonb)` and the down `drop function if exists` is `public.append_audit(jsonb)` with the same arg type. Match is exact. No 42P13 + no ambiguous-arg drop failure.

**Verdict**: C-H1 fix prevents 42P13. Pattern aligns with lesson `patches-operacionales-emergentes-durante-apply` patch #2. The comment at lines 30-35 of the down migration explicitly cites the lesson + C-H1 origin + 42P13 error code. Auditable.
