### B3 — GRAVE: el guard NO captura el caso que el script mide

**Severity:** HIGH (the writers no van a fire en el caso real).

**Evidence:**

`scripts/observe-rls-burn-readiness.ts:190,198`:

```
audit_log?action_type=eq.claim_missing&created_at=gt.<windowStartIso>
audit_log?action_type=eq.active_tenant_null&created_at=gt.<windowStartIso>
```

`SPEC.md:53-58`:

> Pre-condition: `custom_access_token_hook` re-enabled in Hakuna prod for >=24h
> continuous with **zero `claim_missing` audit events from real users**.
> Telemetry: count `claim_missing` **fail-closed events**.

El SPEC define `claim_missing` como **"fail-closed events"** — eso es el caso
en que el hook fallo y Supabase rechazo el mint **antes** de que el usuario
llegue a Next.js. **El guard del Pass 1 corre cuando el usuario YA esta
autenticado y llega a `/app/*`.** Si el hook fallo cerrado, el JWT nunca se
mintea → usuario nunca tiene cookie valida → nunca llega a `requireActiveTenant()`
→ nunca emit `claim_missing`.

Esto es un **gap semantico fundamental**:

| Caso real                                                | Sintoma observable                      | Donde detectar                                  |
| -------------------------------------------------------- | --------------------------------------- | ----------------------------------------------- |
| Hook crashea durante mint                                | Supabase Auth returns 500 a /verify-otp | API route `/auth/verify-otp` o callback handler |
| Hook returns sin `active_tenant_id` (cambio de contrato) | JWT mint OK pero sin claim              | Middleware/guard del Pass 1                     |
| Hook returns `active_tenant_id: null`                    | JWT mint OK con claim null              | Middleware/guard del Pass 1                     |

El Pass 1 cubre **caso 2 y 3**, no caso 1. El **caso 1 es lo que dice "fail-closed"**
en SPEC. Si la metrica `claim_missing` se interpreta literal segun SPEC, el
Pass 1 NO la genera nunca.

**Two readings posibles del SPEC:**

(a) **SPEC interpretation strict:** `claim_missing` = "Auth rechazo mint por
crash del hook" → detect en `/login` flow o en `signInWithOtp` response.
El Pass 1 no cubre esto.

(b) **Implicit redefinition:** `claim_missing` = "JWT existe pero sin claim"
(lo que el Pass 1 cubre). En este caso el SPEC line 57 "fail-closed" es
informal — fail-closed = "no entra a tenant data", lo cual el guard hace via
`redirect("/login?error=...")`.

**Action required pre-code:** ASK Senior PM + Security Engineer **a cual de
las dos lecturas se va.** El doc Pass 1 asume (b) sin discutirlo. Si la
respuesta es (a), el diseno entero cambia — necesita emit-on-mint, no
emit-on-request. Si la respuesta es (b), agregar al doc una nota explicit
clarificando que **"fail-closed" del SPEC line 57 se reinterpreta como
"sad-path redirect"** y que el writer NO cubre el caso de mint-crash (que
seria detectable solo via Supabase Auth logs / Sentry).

Adicionalmente: el guard del Pass 1 emite con `redirect()` despues. Pero
**`redirect()` en Next.js throws `NEXT_REDIRECT` que es caught upstream**. Si
`writeAuditEvent` throwea (DB down), va a propagar **antes** del redirect →
el usuario ve un 500 en lugar de redirect-to-login. Esto es OK por ADR-0007
("audit failure aborts action") pero hay que **decirlo explicito** porque el
caller esperaria un redirect.

### B4 — GRAVE: `audit.read` infinite loop si guard se mete en `/api/audit`

**Severity:** HIGH (recursive audit storm).

**Evidence:** `src/app/api/audit/route.ts:73-79` (main):

```typescript
try {
  await writeAuditEvent({
    action: "audit.read",
    actor_user_id: user.id,
    acting_as_tenant_id: tenantId,
    ...
  });
} catch (e) {
  console.error("[audit.route] meta-audit write failed (non-blocking):", e);
}
```

Pass 1 §2.2.C dice: "agregar `await requireActiveTenant()` al GET handler de
`/api/audit/route.ts`". Esto crea un ciclo:

1. User authenticated llega a `/api/audit?tenant=xxx`
2. `requireActiveTenant()` corre → JWT tiene claim → returns tenantId OK
3. GET handler retorna `audit_log` rows
4. Handler escribe meta-event `audit.read` via `writeAuditEvent`
5. Resultado: 0 events extra (caso happy)

Pero ahora el caso sad-path con `requireActiveTenant()` instalado:

1. User llega a `/api/audit` con JWT sin claim
2. `requireActiveTenant()` emit `claim_missing` via `writeAuditEvent`
3. `redirect("/login?error=missing_tenant_claim")` throws `NEXT_REDIRECT`
4. **Pero el redirect viene de un API route**, no de un server component. En
   un API route, `redirect()` de `next/navigation` **no funciona como esperado**
   — Next.js levanta el error pero el response final es un 500 o un 307 con
   body raro. Hay que **devolver `NextResponse.redirect(url)` en API routes**.

El Pass 1 reusa `redirect()` de `next/navigation` para todos los call sites
sin distinguir page vs API route.

**Action required pre-code:**

- Para API routes: distinguish call site, devolver `NextResponse.json({error:
...}, {status: 401})` o `NextResponse.redirect(...)`.
- Para server components / pages: `redirect()` de `next/navigation` esta OK.
- Refactor: `requireActiveTenant()` partido en dos: `requireActiveTenantPage()`
  (throws via `redirect()`) y `requireActiveTenantApi()` (returns Result type
  o throws Response).

## 2. Issues no-bloqueantes (nice-to-have, post-implementacion)

### N1 — `getSession()` en server component es de-recomendado

**Evidence:** Supabase `@supabase/ssr` v0.10.3 (installed) documenta que
**`getSession()` retorna data del cookie sin verificar firma** en server-side.
La recomendacion oficial es `getUser()` (que SI verifica con `/auth/v1/user`
endpoint).

`active-tenant-guard.ts` Pass 1 hace:

```typescript
const { data: { user } } = await supabase.auth.getUser();   // verifies
...
const { data: { session } } = await supabase.auth.getSession();   // cookie-only
const claims = decodeJwtPayload(session?.access_token);
```

El doble call (`getUser` + `getSession`) es redundante (extra latency ~50-100ms).
Mejor patron: `getUser()` retorna el user; los claims del JWT pueden leerse
de **`user.app_metadata`** o de **`user.aud`** directamente sin re-decode.
PERO `active_tenant_id` esta en JWT root claims, no en `user_metadata` ni en
`app_metadata`. Entonces decode manual sigue siendo necesario.

**Sugerencia:** dropear `getSession()`. Leer cookie directamente con
`cookies().get('sb-<ref>-auth-token')`, decode payload. Una RPC menos.
Tradeoff: codigo mas brittle a cambios de cookie naming. **Track post-impl.**

### N2 — `redirect()` con error param fuga info en logs

`redirect("/login?error=missing_tenant_claim")` y derivados van a aparecer
en Vercel Edge logs + en browser referrer (si user navega adelante con header
`Referer`). El error code en URL filtra detalle al user finalshell — fingerprint
del sistema (un atacante sabe que existen multiples error codes que mapean a
modos de falla distintos).

**Sugerencia:** colapsar todos los redirects a un solo `?error=auth` y
**mantener el detalle en el audit_log row**. El user ve "auth failed",
internal teams ven el detalle via audit log. Track post-impl.

### N3 — `metadata.jwt_claim_keys` puede contener PII en el futuro

Pass 1 emite `metadata.jwt_claim_keys: Object.keys(claims)`. Hoy, Supabase
JWTs tienen un set conocido (`iss, aud, sub, email, role, ...`). Si Supabase
agrega claims con nombres derivados de user data (improbable, pero un custom
hook futuro podria), los **nombres** caen en `metadata`. Hoy: not an issue.
Futuro: posible. **Sugerencia:** allowlist explicito de claim names a
loguear, drop el resto. Track post-impl.

## 3. Open questions del Pass 1 que NO resuelve, y que yo TAMPOCO resuelvo

### OQ-P1.4 — Script fix scope (Pass 1 line 372)

"Confirmar que arreglar `observe-rls-burn-readiness.ts` columnas es parte de
W1.T2 (mi posicion) vs separar en W1.T2b."

**Mi posicion como cold reviewer:** **separar en W1.T2b**. Razones:

1. **Diferent semantics:** W1.T2 son **writers** (instrumentation). El script
   es **consumer** (gate readiness). Mezclar significa que un revert del
   writer-side rompe el script.
2. **Diferent reviewers:** writers Pass 1+2 son BA. Script changes deberian
   pasar por Security Engineer + BA (porque el script es el que dispara el
   GO/NO-GO de la burn migration).
3. **Atomic deploy:** writers pueden mergearse sin el script (no breakage —
   solo "no rows aparecen"). Script fix sin writers tambien (script lee 0 →
   verdict NO-GO → safe). Pero el bundle puede crear un caso donde algo del
   bundle revierte y el otro no.

**Tradeoff:** dos PRs pequenos vs uno mediano. Para Two-Pass review extended,
dos PRs es mejor.

### OQ-P1.5 — Request-scope dedup (Pass 1 line 373)

"Mi diseno NO deduplica emits dentro de un request. Si el guard se invoca 3
veces en un render tree, emite 3 rows."

**Mi posicion como cold reviewer:** **dedup via `cache()` de React.** Razones:

1. **Hash chain serialization:** 3 emits = 3 advisory locks acquired
   secuencial → 3 round trips DB. En Hakuna baja carga es nada. En
   tenant futuro = 3x latency hot path.
2. **Signal noise:** el burn-gate cuenta filas. 3 rows desde la misma
   request inflan el contador artificially. El script tiene heuristic
   "instrumentationGap = tokenMints>0 AND anyRow.length===0" que no
   detecta el inverso ("inflated rows from dedup miss").
3. **Detection de retry storms** (argumento pro-no-dedup del Pass 1): es
   detectable independientemente con `count(*) group by minute` sobre
   `audit_log`. No necesita per-request dup.

Sugerencia: usar `import { cache } from 'react'` wrapping
`requireActiveTenant`. Implementacion una linea, tradeoff favorable.
