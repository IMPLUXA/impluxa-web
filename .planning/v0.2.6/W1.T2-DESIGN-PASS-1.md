## 2. Diseno de writers faltantes

### 2.1 Decision arquitectural: emitir desde Next.js, NO desde PL/pgSQL

| Opcion                                                                                                                       | Pro                                                                                                                                                                                                                         | Contra                                                                                                                                                                                                                                                                                                                                                                                              | Veredicto  |
| ---------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- |
| **A. Emitir desde el hook PL/pgSQL** (`custom_access_token_hook` hace INSERT a `audit_log` cuando `v_active_tenant is null`) | Captura el caso al momento exacto del mint, evidencia DB-side definitiva                                                                                                                                                    | (1) Hook corre como `supabase_auth_admin`, NO como `service_role`. `append_audit` esta granted SOLO a `service_role` (...005_audit_log.sql:131). Hook no puede llamarla. (2) Hash chain trigger usa `pg_advisory_xact_lock` — meterlo dentro del hook serializa los mints. (3) Hook es `stable` + fail-closed; INSERT lo hace `volatile` y rompe contract. (4) Mas grants nuevos = mas blast radius | **REJECT** |
| **B. Emitir desde el middleware Next.js** + un guard server-side nuevo (`requireActiveTenant()`)                             | (1) Reusa `writeAuditEvent` existente con service-role. (2) Captura el evento donde se observa el sintoma (request handler ve el JWT decoded). (3) No toca hook ni grants DB. (4) Test coverage facil (unit + integration). | (1) Si user llega a un endpoint que no esta protegido por el guard, no se emite (cobertura parcial). (2) Latency: 1 RPC extra en cada fail-closed branch (~50ms p50)                                                                                                                                                                                                                                | **ACCEPT** |
| C. Hibrido: hook emite `active_tenant_null` (single row, via `dblink` o `notify`), middleware emite `claim_missing`          | Captura ambos eventos donde son detectables.                                                                                                                                                                                | Complejidad cruzada (2 mecanismos, 2 test stacks). `dblink` aumenta superficie. `pg_notify` no es auditable.                                                                                                                                                                                                                                                                                        | **REJECT** |

**Veredicto:** Opcion **B**. Razones:

1. No requiere modificar el hook v0.2.5 (que esta re-enabled hace <48h y es activo en prod).
2. Reusa `writeAuditEvent` 1:1 (mismo signature que `email-hook` y `tenant/switch` usan).
3. Failure mode: `writeAuditEvent` throws → middleware throws → request 500. Esto es **deseado** per ADR-0007 ("a silently failed audit write is worse than the action being aborted").
4. Hash chain monotonicity preservada (writes serializadas via advisory lock DB-side, ortogonal a quien las inicia).

### 2.2 Files to add

#### A. `src/lib/audit-log/burn-readiness-events.ts` (new, ~60 LOC)

```ts
import "server-only";
import { writeAuditEvent } from "@/lib/auth/audit";

/**
 * v0.2.6 W1.T2 — burn-readiness telemetry writers.
 *
 * Both emit via the existing v0.2.5 `writeAuditEvent` → `public.append_audit`
 * RPC path (service_role, hash chain preserved). Errors propagate (do NOT
 * swallow — ADR-0007 monotonicity invariant).
 *
 * Consumer: `scripts/observe-rls-burn-readiness.ts` reads these rows over the
 * 24h pre-burn window. Both action strings are stable identifiers — do not
 * rename without updating the script.
 */

/** Emit when a JWT reaches a protected endpoint without `active_tenant_id` claim. */
export async function emitClaimMissing(args: {
  userId: string | null;
  ip?: string;
  userAgent?: string;
  requestId?: string;
  jwtClaims: Record<string, unknown>; // sanitized summary, NEVER the raw token
}): Promise<void> {
  await writeAuditEvent({
    action: "claim_missing",
    actor_user_id: args.userId ?? undefined,
    ip: args.ip,
    user_agent: args.userAgent,
    request_id: args.requestId,
    metadata: {
      jwt_claim_keys: Object.keys(args.jwtClaims), // names only, no values
      iss: typeof args.jwtClaims.iss === "string" ? args.jwtClaims.iss : null,
      aud: typeof args.jwtClaims.aud === "string" ? args.jwtClaims.aud : null,
    },
  });
}

/** Emit when JWT has `active_tenant_id` key but its value is null/empty (hook misfire). */
export async function emitActiveTenantNull(args: {
  userId: string;
  ip?: string;
  userAgent?: string;
  requestId?: string;
}): Promise<void> {
  await writeAuditEvent({
    action: "active_tenant_null",
    actor_user_id: args.userId,
    ip: args.ip,
    user_agent: args.userAgent,
    request_id: args.requestId,
    metadata: { hook_misfire: true },
  });
}
```

**Notes on payload safety:**

- NO logueamos el JWT crudo en `metadata` (PII / token). Solo `Object.keys()`.
- `iss` y `aud` son safe (constantes de Supabase, no secrets).
- `ip` cae en columna `inet` (validada por Postgres, no inyectable).

#### B. `src/lib/auth/active-tenant-guard.ts` (new, ~80 LOC)

```ts
import "server-only";
import { redirect } from "next/navigation";
import { headers } from "next/headers";
import { getSupabaseServerClient } from "@/lib/supabase/server";
import {
  emitClaimMissing,
  emitActiveTenantNull,
} from "@/lib/audit-log/burn-readiness-events";

/**
 * Returns the `active_tenant_id` UUID from the JWT for the current request, or
 * triggers an audit event + redirect when the claim is missing/null.
 *
 * Called from tenant-scoped page server components (app/tenant/[slug]/*) and
 * tenant-scoped API routes. Idempotent: safe to call twice in one request
 * (each call emits one row — by design, so we can detect retry storms).
 *
 * For request-level dedup, callers SHOULD cache the return value in the request
 * scope (Next.js `cache()` helper) — not implemented here to keep instrumentation
 * write-path-equal regardless of caller pattern.
 */
export async function requireActiveTenant(): Promise<string> {
  const supabase = await getSupabaseServerClient();
  const {
    data: { user },
  } = await supabase.auth.getUser();
  if (!user) redirect("/login");

  // Read the JWT claims via Supabase server client (decoded automatically).
  const {
    data: { session },
  } = await supabase.auth.getSession();
  const claims = (
    session?.access_token ? decodeJwtPayload(session.access_token) : {}
  ) as Record<string, unknown>;

  const h = await headers();
  const ip = h.get("x-forwarded-for")?.split(",")[0]?.trim() ?? undefined;
  const ua = h.get("user-agent") ?? undefined;
  const reqId = h.get("x-request-id") ?? undefined;

  if (!("active_tenant_id" in claims)) {
    await emitClaimMissing({
      userId: user.id,
      ip,
      userAgent: ua,
      requestId: reqId,
      jwtClaims: claims,
    });
    redirect("/login?error=missing_tenant_claim");
  }

  const value = claims.active_tenant_id;
  if (value === null || value === "" || value === undefined) {
    await emitActiveTenantNull({
      userId: user.id,
      ip,
      userAgent: ua,
      requestId: reqId,
    });
    redirect("/login?error=null_tenant_claim");
  }

  if (typeof value !== "string") {
    // Defensive: hook contract requires string UUID. Treat as null.
    await emitActiveTenantNull({
      userId: user.id,
      ip,
      userAgent: ua,
      requestId: reqId,
    });
    redirect("/login?error=invalid_tenant_claim");
  }

  return value;
}

/** Minimal JWT payload decoder. Does NOT verify signature (Supabase already did). */
function decodeJwtPayload(jwt: string): unknown {
  const parts = jwt.split(".");
  if (parts.length !== 3) return {};
  try {
    const b64 = parts[1].replace(/-/g, "+").replace(/_/g, "/");
    return JSON.parse(Buffer.from(b64, "base64").toString("utf-8"));
  } catch {
    return {};
  }
}
```

**Why server-component-call-site and not middleware:**

- Next.js middleware runs on Edge runtime → `getSupabaseServerClient()` (which uses cookies) is awkward + adds latency to EVERY request including static assets.
- Server components run on Node + already touch cookies. Per-request cost is zero-overhead for non-tenant routes.
- Middleware change risk = host rewriting regression. Avoided.
