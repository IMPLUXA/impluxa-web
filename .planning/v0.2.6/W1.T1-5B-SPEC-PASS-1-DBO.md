## 2. DB invariants

### DI1 — Atomic audit_log append + hash chain

`append_audit` corre en single transaction. Hash chain trigger `audit_log_compute_hash` (BEFORE INSERT) computa `prev_record_hash` + `record_hash` antes del physical INSERT. Si trigger falla (advisory lock timeout, sha256 failure unlikely) → INSERT rollbacks → ningun audit_log id se incrementa (bigserial). Concurrent appends serializados via `pg_advisory_xact_lock(hashtext('audit_log_chain'))` (005:58). **No partial commits posibles.**

Evidence: `audit_log_compute_hash` body 005:47-89; advisory lock semantics ADR-0007.

### DI2 — `audit_dedup` uniqueness (jwt_jti, action)

PK compuesta `(jwt_jti, action)` (file 001:24) garantiza max 1 row per pair. `INSERT ... ON CONFLICT DO NOTHING` (file 001:144-146) + `GET DIAGNOSTICS v_claimed = row_count` (file 001:148) → si dedup ya existia, row_count=0, function short-circuit return null. Combinado con CHECK constraint `action IN ('claim_missing','active_tenant_null')` (file 001:25), solo los 2 event-types tracked pueden insertar dedup rows. Cross-pod / cross-restart atomic gate.

### DI3 — Cron rotation double-buffer creates next BEFORE current fills

`audit_log_rotate_partitions()` loop `for v_month in 1..2` (006:32) crea `current_date + 1 month` Y `current_date + 2 months`. Si cron schedule es dia 25 de cada mes (`0 0 25 * *` per migration comment 006:4), al correr 2026-07-25 crea `audit_log_2026_08` Y `audit_log_2026_09`. Particion del mes corriente (07) ya existe. **Una corrida fallida no rompe el sistema** — la proxima corrida sigue creando ambas (idempotent `if not exists` check 006:37-44).

**Caveat operacional:** pg_cron extension habilitacion en prod aun pendiente (verified db-second-pass §E3 + observability gate). Hoy las partitions hasta 2026-08-01 fueron creadas por `select public.audit_log_rotate_partitions()` manual en migration body (006:64). Si pg_cron no se habilita pre-2026-07-15 (F2 deadline db-second-pass), inserts agosto pueden fallar con `no partition of relation "audit_log" found for row`. **Cubierto en Seccion 4 fuera-de-scope-5.B + Seccion 5 OQ.**

### DI4 — `app_config.key` uniqueness via PK

`key text PRIMARY KEY` (app_config:36). Cualquier INSERT con key duplicada falla con unique violation. UPSERT pattern operacional: `INSERT ... ON CONFLICT (key) DO UPDATE SET value = excluded.value, updated_at = now()`. **No hay UPSERT automatico en migration ni en runtime code** — operator manual.

### DI5 — Function return type changes need `drop function if exists` precedent

Operational invariant codificado por lesson `patches-operacionales-emergentes-durante-apply` (s10a). Postgres rechaza `CREATE OR REPLACE FUNCTION` cuando cambia el `RETURNS <type>`. Patron correcto (aplicado en file 001:111):

```sql
drop function if exists public.append_audit(jsonb);
create or replace function public.append_audit(p_event jsonb) returns bigint ...
```

Migration debe ejecutar ambos statements en la misma transaccion para no dejar ventana sin function. Aplicado en migration `001` que es file-level transactional por default Supabase migration runner.

### DI6 — Hash chain payload format frozen across event-types

Hash payload (005:67-79): `coalesce(prev_hash, '') | occurred_at | actor_user_id | actor_session_id | acting_as_tenant_id | acting_as_role | action | resource_type | resource_id | ip | user_agent | request_id | metadata`. **Cualquier columna nueva agregada al table debe agregarse al payload Y todos los recomputers downstream** (ADR-0007 baseline). Cut B no agrega columnas → invariant holds.

### DI7 — `audit_dedup` does NOT participate in hash chain

`append_audit` solo inserta en audit_log si pasa el dedup gate. Dedup rows son **transient observability state** (7-day TTL by design, comment 001:13-15 down migration). No participan en hash chain. Backup/restore puede TRUNCATE audit_dedup sin afectar audit_log integrity.

### DI8 — Cron jobs are idempotent across migration re-runs

`cron.schedule` raises "job already exists" si duplicate. Migration usa `cron.unschedule(...)` previa wrapped en `BEGIN ... EXCEPTION WHEN OTHERS THEN null` (001:62-67 fix B9.1). Re-runs de la migration **update cron expression** correctamente. Sin esto, edits al cron body NO se reflejaban (silent bug).

### OQ-DB-2 — `hook_reenable_ts` initial value UPSERT

**Pregunta:** ¿Como se popula el row `app_config(key='hook_reenable_ts', value=<timestamp>)`? Migration NO lo inserta. ¿Operator manual via SQL editor / runbook step?
**Lean firme:** Documentar en runbook v0.2.6 (W3 observability section) que **operator ejecuta UPSERT en el momento exacto de re-habilitar el hook**, con SQL exacto:

```sql
insert into public.app_config (key, value, updated_at, updated_by)
values ('hook_reenable_ts', to_jsonb(now()::text), now(), 'operator:<email>')
on conflict (key) do update set value=excluded.value, updated_at=now(), updated_by=excluded.updated_by;
```

**Risk:** MEDIUM. Sin esto, `observe-rls-burn-readiness.ts` no tiene anchor y cae a fail-closed NO-GO (intencional per script:90 comment). Pero el operador puede olvidar y bloquear el burn 24h gate sin razon. **Mitigation:** runbook step explicito + Telegram ASK pre-hook-reenable.

3. Test fallback si key missing (default 7d).
