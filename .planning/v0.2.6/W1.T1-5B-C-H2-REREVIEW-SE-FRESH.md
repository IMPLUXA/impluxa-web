## Question 1 — Threat model

Under what threat would a tenant-claim action JWT have missing JTI? Per-scenario reachability in production:

### S1 — Compromised auth issuer forges a token without JTI

**Reachable in prod?** NO.
**Evidence:** `guard.ts:104-115` `extractJti(accessToken)` runs **after** `requireUser()` (L172) which calls `supabase.auth.getUser()`. That call validates the token against gotrue with the SUPABASE-managed signing keys (`runtime-config.ts:38` `SSO_JWT_SECRET`). A forger without the signing key cannot produce a token that gotrue accepts; if they obtained the key, JTI-stripping is the least of our problems — they own the whole auth surface. **Threat model out-of-scope** (covered by ADR-0005 §3 / SE-H1 service_role grant posture).

### S2 — Auth library bug: token rotation race produces token without JTI claim

**Reachable in prod?** YES — this is the only realistic path.
**Evidence:** gotrue's refresh-token rotation issues a fresh access_token. The JTI claim is populated by gotrue's mint code; if a future Supabase SDK upgrade or gotrue version introduces a code path that mints without JTI (regression, feature flag, hook output that strips claims), tokens reach `guard.ts` with `payload.jti === undefined`. `decodeJwt(accessToken)` returns the payload as-is — `extractJti` returns `undefined` on L114 → flows to `writeAuditEvent` with `jwt_jti: undefined` → `audit.ts:86-98` warn → RPC continues → SQL `v_jti = nullif(p_event ->> 'jwt_jti', '')::uuid` is null at L136 → dedup gate condition `v_jti is not null` at L143 fails → fallthrough to direct INSERT at L154. **Every retry = new row.**

This is a real path. But it requires upstream regression to trigger.

### S3 — Replay attack with stale token from before JTI was added

**Reachable in prod?** NO at v0.2.6 baseline.
**Evidence:** v0.2.5 already had `custom_access_token_hook` emitting tokens; JTI has been a standard claim across gotrue versions in the codebase's history. Any stale token old enough to lack JTI would also be past TTL (1h per Supabase config) and rejected by gotrue. The only way a stale-no-JTI token reaches `guard.ts` is if gotrue accepts it — see S1.

### S4 — Internal call path constructs synthetic payload omitting JTI

**Reachable in prod?** NO at app code-path level; YES at test-code-path level.
**Evidence:** Grep for `writeAuditEvent` call sites: only `guard.ts:134` (`emitTenantClaimAudit`) and `guard.ts:34` (`emitApprovalGateBypassAudit`) in `src/`. Both source `jwt_jti` from `extractJti(session?.access_token)` where `session` comes from `supabase.auth.getSession()` — a real auth-validated session. No synthetic payload path in production code. Test code (`tests/unit/lib/audit.test.ts` D2) does construct synthetic payloads, but tests don't reach the gate.

### S5 (added) — Operator-side seed / synthetic test event in prod

**Reachable in prod?** YES, but per OQ-5 locked decision, "synthetic test events do NOT reset the clock" — operator-emitted synthetic events are explicitly excluded from FR-RLS-BURN-2 count. So even if such events fire JTI-null and write multiple rows, they're filtered upstream by event provenance (script-side filter on `actor_user_id` not in real-user set).

### Summary

The only realistic in-prod path is **S2 (gotrue/SDK regression)**. Probability over the 24h observability window: low but non-zero (Supabase ships SDK updates regularly; we don't pin gotrue version explicitly).

## Question 2 — Impact on flip gate

### Severity classification

**MED, not HIGH.** Rationale:

**HIGH would require:** a single attacker can trigger N JTI-stripped retries to flip the gate to a **wrong reading** (false GO → flip `hakuna_live=true` when system is unhealthy).

**Actual impact:** the dedup bypass inflates the `claim_missing` count. Counter goes UP, not DOWN. Per FR-RLS-BURN-2 acceptance criterion (SPEC L50-60): `claim_missing` "must trend to zero or stay zero". An inflated count makes the gate **less likely to fire as ready**, not more. False NO-GO direction = **fail-closed** = the safe direction.

**Confidentiality:** untouched — no PII leaks, no privilege escalation.
**Integrity:** untouched — hash chain is preserved (every insert appends, hash chain trigger fires per row, monotonic).
**Availability of the gate (the actual concern):** noise corrupts a _readiness_ signal, but never to FALSE GO. The operator sees an inflated count and re-investigates — exactly the intended observability behavior. The blast radius is one round-trip of human gating, not a security incident.

**Conditions where this becomes HIGH:**

- If FR-RLS-BURN-2 acceptance flipped to "ignore claim_missing if count is below threshold X" instead of "must be zero". Then an attacker could pad noise _below_ threshold while a real `claim_missing` bug also runs _below_ threshold → false GO. **Current acceptance is `== 0` (per OQ-5), so this hypothetical is closed.**
- If `claim_missing` were used to LOCK OUT users (e.g., automated kill switch on threshold). Current code does NOT do this — the gate is operator-facing only.

### Auth/input validation before `audit.ts`?

YES. Walking the call chain backward from `audit.ts:86`:

1. `guard.ts:134` (`emitTenantClaimAudit`) — invoked only inside `requireActiveTenantOrRedirect` (L187) and `requireActiveTenantOrResponse` (L231) after `readActiveTenantClaim(user)` returns `!ok` (L182, L227).
2. `readActiveTenantClaim` (L84-92) reads `user.app_metadata.active_tenant_id`. `user` comes from `requireUser()` → `supabase.auth.getUser()` — gotrue-validated.
3. So before reaching audit, the JWT was gotrue-validated. **JTI presence is NOT validated by gotrue** — JTI is a Supabase-internal claim, gotrue doesn't enforce its presence on incoming tokens (it generates it on mint, but doesn't require it on accept).
4. `extractJti` (L104-115) reads `payload.jti` and returns `undefined` if absent/malformed. No rejection.

**Why doesn't auth boundary reject JTI-null?** Because historically the system tolerated JTI absence (T5 backward-compat noted in `audit.ts:35-36` "absent JTI falls back to the legacy write-every-time path"). Rejecting at the auth boundary would be a behavioral break — sessions minted before v0.2.5 hook deployment, or tokens minted during a brief hook outage, would 401 the user out. Operationally that's worse than gate-noise.

### Is the dedup gate a security control or a perf optimization?

**It is a SIGNAL CORRECTNESS control, not a security control and not pure perf.**

- **Not security:** does not enforce isolation, authz, or rate limits. If you removed it, no boundary is breached — `audit_log` just gets noisier. The actual security control is `audit_log` itself being service_role-only (SE-H1) and the hash chain being tamper-evident.
- **Not pure perf:** the rationale in `audit_dedup.sql:1-16` is "without this, two concurrent requests with the same JWT insert duplicate rows, **inflating the hash chain with noise and breaking the gate signal**". The intent is _gate signal correctness_, not throughput.
- **Therefore:** the dedup gate is a **readiness-signal hygiene control**. It can warn-and-continue (current behavior) without violating security posture. Its failure mode (warn-only when JTI absent) degrades signal quality, not security.

Different rule per category: security controls fail-closed. Perf gates can warn-continue. Readiness-hygiene controls can warn-continue **IF the downstream consumer of the signal is robust to noise**. That's the lever — the fix lives at the consumer (script), not at the producer (audit.ts).

## Question 3 — Verdict

### DEFER-W1.T2 with explicit tripwire

**Rationale:**

1. **Not a security bug.** No exploitable boundary breach. Worst case is operator-facing signal noise that fails closed (no false GO on the gate).
2. **Primer pass is correct that warn-only is acceptable semantic.** The warn surfaces the operational signal; that's the mitigation at the producer side.
3. **Cold pass is correct that the gate signal can be corrupted in S2 scenarios.** But the corruption direction is safe (NO-GO) and the right fix is at the consumer (script-side collapse), not at the producer.
4. **Shipping a producer-side fix (fail-closed at audit.ts, or synthetic dedup key) introduces NEW risk:**
   - Fail-closed at audit.ts would block the redirect/response on a JTI-missing event — _breaking real users_ during a gotrue regression. The bug becomes user-visible instead of operator-visible. Worse posture.
   - Synthetic dedup key (`actor_user_id + day_bucket` PK in audit_dedup) requires migration + CHECK constraint update — schema change in Cut B. Out-of-scope per the truncado intent.
5. **W1.T2 is the right home** because W1.T2 is the audit-log writers wave; the script-side filter (`observe-rls-burn-readiness.ts`) is already the W1.T2 deliverable per cold-pass OQ-5B.8.

     (d) seed mixed JTI-present (50 rows, same JTI, dedup gate fires → 1 inserted) + JTI-null (50 rows, same actor + minute) → script reports `claimMissing.uniqueActors == 2` (one for the JTI-present cluster, one for the JTI-null cluster).

5. **No new test required for `audit.ts`** beyond the existing `audit.test.ts:D2` warn coverage (cold-pass already specifies extending D2 to spy on `console.warn`; that's a 5-line addition).

### Why not FIX-AHORA

Considered three FIX-AHORA mitigations and rejected all three:

- **Reject JTI-null payloads at auth boundary (`guard.ts:107-114` change `extractJti` to throw, or `requireUser` to 401 on JTI-null):** breaks T5 backward-compat. Real users get 401 during gotrue regression instead of operator getting a warn. **Higher blast radius than the bug.**
- **Fail-closed at `audit.ts:90` (warn + return without RPC):** drops the audit_log row entirely on JTI-null. Now FR-RLS-BURN-2 can produce **false GO** if a real claim_missing bug coincides with a JTI-null window. **Reverses the failure direction from safe to unsafe.** Net negative.
- **Extend dedup gate to use synthetic key (migration changes `audit_dedup_action_chk` + adds `actor_user_id` to PK):** schema change in Cut B-truncado scope. Out-of-scope per the truncado intent. Belongs in v0.2.7 (matches cold-pass OQ-5B.8 lean: DEFER to v0.2.7).

### Why not RESOLVED

Primer pass treats warn-only as the mitigation. Cold pass correctly notes the script does not yet consume the warn into a collapsed count — so RESOLVED would be premature. The warn alone does nothing for FR-RLS-BURN-2 unless the consumer collapses. The consumer-side collapse is the actual closure, and it ships in W1.T2.

### Why not NEEDS-RUNTIME-VERIFICATION

No runtime experiment changes the verdict. The semantics are visible in code (`audit.ts:86-98` + `audit_dedup.sql:143`). Cold-pass already traced the chain accurately. Runtime test belongs to W1.T2 acceptance (item 4 of tripwire above).
