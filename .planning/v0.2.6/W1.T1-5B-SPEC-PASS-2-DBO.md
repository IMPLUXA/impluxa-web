LIMIT 10000 ... exit when not found end loop` (PL/pgSQL EXIT-driven, no transaction control needed; relies on single-transaction lock on table being acceptable for ~200K rows). ## 4. pg_cron deadline 2026-07-15 — mathematical derivation

### Inputs

- Migration `005_audit_log.sql:34-36`: initial partition `audit_log_2026_05 for values from '2026-05-01' to '2026-06-01'`.
- Migration `006_audit_partition_rotation.sql:64`: smoke `select public.audit_log_rotate_partitions()` runs on apply (one-time).
- Migration `006:32-56`: rotation function loops `for v_month in 1..2`, creating partitions for `current_date + 1 month` and `current_date + 2 months`.
- Migrations applied 2026-05-14 (timestamp in filename) per `git log`.
- Cron schedule per migration comment (006:4): `0 0 25 * *` (day 25 of each month).

### Derivation

**Smoke at apply (2026-05-14)** with `current_date = 2026-05-14`:

- `v_month=1`: `current_date + 1 month = 2026-06-14` → `date_trunc('month', ...) = 2026-06-01` → creates `audit_log_2026_06 for values from '2026-06-01' to '2026-07-01'`.
- `v_month=2`: `current_date + 2 month = 2026-07-14` → `date_trunc('month', ...) = 2026-07-01` → creates `audit_log_2026_07 for values from '2026-07-01' to '2026-08-01'`.

**Result post-apply state (2026-05-14):**

- `audit_log_2026_05` covers `[2026-05-01, 2026-06-01)` ✓
- `audit_log_2026_06` covers `[2026-06-01, 2026-07-01)` ✓
- `audit_log_2026_07` covers `[2026-07-01, 2026-08-01)` ✓
- **No partition covers `[2026-08-01, 2026-09-01)`.**

**First insert failure date = `occurred_at >= '2026-08-01 00:00:00 UTC'`** with error:

```
ERROR:  no partition of relation "audit_log" found for row
DETAIL:  Partition key of the failing row contains (occurred_at) = (2026-08-01 00:00:00+00).
```

### pg_cron deadline calculation

If pg_cron is enabled and the cron job `audit_log_rotate_partitions` runs on day 25 each month:

- **2026-06-25 tick**: `current_date = 2026-06-25` → creates `2026_07` (already exists, idempotent skip) + `2026_08`. ✓
- **2026-07-25 tick**: creates `2026_08` (skip) + `2026_09`. ✓
- **2026-08-25 tick**: creates `2026_09` (skip) + `2026_10`. ✓

If pg_cron is enabled by **2026-06-25** → `2026_08` gets created during the June tick → August inserts succeed. ✓

If pg_cron is enabled between 2026-06-26 and 2026-07-24 → the next tick on 2026-07-25 creates `2026_08` → August inserts succeed (with shorter buffer). ✓

If pg_cron is enabled **after 2026-07-25 but before 2026-08-01** → still safe ONLY IF an operator manually runs `select public.audit_log_rotate_partitions();` as part of the enable. ⚠️ Risk: enable without manual smoke = no `2026_08` partition.

If pg_cron is enabled **on or after 2026-08-01** → first insert on/after 2026-08-01 fails until operator runs the manual smoke. **Production downtime risk.**

### Recommended deadline

**Hard deadline (safe with buffer): 2026-07-24** (one calendar day before the cron tick on 2026-07-25, gives July tick the chance to create `2026_08`).

**Soft deadline (no buffer): 2026-08-01** (with mandatory operator-run smoke at enable time).

Pass 1's `2026-07-15` deadline is more conservative than mathematically required, but the conservatism is **operationally sound** because:

- Allows 10-day buffer for hand-off completion + operator scheduling.
- Catches cases where Supabase Dashboard enable + first cron-worker spawn has propagation delay.
- Allows time for any retroactive partition creation if needed.

**DBO Pass-2 endorses 2026-07-15** as the planning deadline, while noting that **2026-07-24** is the hard fallback. PR #6 description should cite both, with 2026-07-15 as the operational target and 2026-07-24 as the **point of no return**.

## 5. D-COLD-4 methodology — copy-pasteable block for runbook

Include this verbatim in `PLAN.md` or `RUNBOOK-5B.md`:

```markdown
### D-COLD-4 — pg_inherits partition index propagation re-verify

**When**: any time `pg_get_indexdef` output for a partitioned table parent index
returns text containing `ON ONLY`, raising suspicion that child partitions are
missing the index.

**Why not trust `pg_get_indexdef`**: PG17 changed the textual representation of
partitioned indexes to display `ON ONLY` even when child partitions DO inherit
the index automatically. Misleading. The actual propagation lives in
`pg_inherits` between the parent index oid and child index oids.

**Canonical re-verify SQL** (run as service_role or postgres):

    select c.relname as child_idx,
           p.relname as parent_idx,
           pn.nspname as namespace
    from pg_index i
    join pg_class c on c.oid = i.indexrelid
    join pg_inherits inh on inh.inhrelid = c.oid
    join pg_class p on p.oid = inh.inhparent
    join pg_namespace pn on pn.oid = p.relnamespace
    where p.relname in (
      'audit_log_acting_tenant_occurred_idx',
      'audit_log_actor_user_occurred_idx',
      'audit_log_pkey'
    );

**Pass criterion**: returns `(N_partitions × N_parent_indexes)` rows, each child
linked to its parent. For current state with 3 partitions (2026_05/06/07) and 3
parent indexes: expect 9 rows. If fewer → propagation broken; manual
`create index ... on <partition>` per missing pair.

**PG11+ behavior**: `CREATE INDEX ON parent` (no `ONLY` keyword) auto-propagates
to existing partitions AND auto-attaches on future `CREATE TABLE ... PARTITION OF`.
Standard since PG11. The `ON ONLY` text in PG17 catalog views is presentation
only, not semantic.
```

This block is self-contained, 8-12 logical lines of prose + 1 SQL block + pass criterion. Ready to paste.
