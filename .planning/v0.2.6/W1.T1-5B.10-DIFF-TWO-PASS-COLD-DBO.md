**Recommendation (defer to W1.T2 or post-v0.2.6):** add explicit `begin; ... commit;` to both `20260518_v026_001_audit_dedup.sql` and `20260518_v026_001_audit_dedup_down.sql` to match the `002` posture. Track in BACKLOG.

### MED-2 — D10.7 residual closure criterion is not operationally crisp

**Location:** `docs/adrs/0010-cut-b-truncado-deferred-consumers-and-migration-discipline.md:151` ("Final closure happens at pg_cron enable in prod") and RUNBOOK-5B.md:228 ("Residual closure at pg_cron enable in prod").

The ADR + RUNBOOK both say "closure happens at pg_cron enable in prod" but neither specifies the single empirical check that constitutes closure. A future operator at the moment of enabling pg_cron will not know whether closure means:

- (a) the cron job `audit_dedup_gc` has at least one row in `cron.job_run_details` with `status='succeeded'`?
- (b) the cron body executes the `do $batch$ ... commit; ... end $batch$` pattern at least once without error?
- (c) an arbitrary `do $batch$ commit; end $batch$` runs at top level (the empirical test from s15)?
- (d) something else?

The s15 empirical test (RUNBOOK §5) was (c) — a synthetic top-level DO block, not the actual cron body. The closure at enable time should be (b): after enabling pg_cron + scheduling the cron + running `select cron.schedule(...)` per §3.6, the operator queries `cron.job_run_details` after the first cron tick (or after a `select public.audit_dedup_gc_run_now();` if such a helper exists — it doesn't today) and confirms `status='succeeded'`.

**Why MED:** the gap is operationally absorbable (Claudia or operator at enable time will figure it out), but the ADR section labeled "Revisit trigger" is silent on what counts as the trigger event for closure.

**Recommendation (defer to PR #6 description or runbook §3.6):** add a §3.7 to RUNBOOK-5B.md with the post-enable closure checklist:

```sql
-- Closure check for DBO-H5 + I7 (run after first cron tick post-enable):
select jobname, runid, status, return_message, start_time, end_time
from cron.job_run_details
where jobname = 'audit_dedup_gc'
order by start_time desc
limit 5;
-- Pass criterion: at least one row with status='succeeded'.
```

### E1 — C-H1 fix correctness

**Test:** simulate the up migration final state (`append_audit returns bigint`) and apply the C-H1 pattern (`drop function if exists ... ; create or replace function ... returns void`).

**Setup query** (synthetic function `test_ch1_repro` to isolate from real schema):

```sql
    raise notice 'WITHOUT_DROP: as expected error % %', sqlstate, sqlerrm;

end $$;

-- Step 3: apply the C-H1 fix pattern:

```

**Result:**

```

```

**Verdict:** C-H1 fix is empirically VERIFIED on PG 17.6 Supabase managed. The `drop function if exists` precedence allows the return-type change to apply cleanly. Step 2's control DID raise 42P13 (notice channel suppressed in MCP output but no error in step 3's final result), confirming the fix is necessary AND sufficient.

**Concurrency note (related to MED-1):** if a long-running RPC call against `public.append_audit(jsonb) returns bigint` is in-flight at the moment `drop function if exists` runs, PG holds an `AccessExclusiveLock` request which queues behind the in-flight call. The drop waits until the in-flight call completes (does NOT error). After the drop, new RPC calls see `42883 undefined_function` until the `create or replace function` runs. With `001_down.sql` running in a single implicit transaction, the window is the time between the two statements within the transaction — sub-millisecond on a quiescent system. On a busy prod system at the moment of rollback, callers in flight see no impact, but callers arriving in the sub-ms window get 42883. Operationally acceptable for a manual rollback context. Recorded in MED-1.
