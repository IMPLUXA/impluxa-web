## 1. Feature summary

Cut B-truncado lands the **app-layer kill switch + readiness scaffolding** of v0.2.6 W1.T1 without yet shipping the W1.T2 audit-log writers or the RLS burn migrations. Three concrete deliveries:

1. **APPROVAL_GATE kill switch (ADR-0005 §5 break-glass).** A module-load-frozen env var (`APPROVAL_GATE_ENABLED=0`) toggles both tenant-claim guard entrypoints from "enforce claim or reject" into "let the request through with a sentinel tenant id and a forensic audit row". This unblocks operator recovery after a regression in v2 RLS policies or hook misfire without requiring a code-redeploy. The bypass is deliberately fail-loud at the data layer: the sentinel tenant id `"__bypass__"` does not match any real tenant row, so RLS still denies tenant-scoped reads.

2. **`public.app_config` skeleton table.** A `(key text primary key, value jsonb, updated_at, updated_by)` key-value store, RLS-enabled with **no policies = deny-all** + explicit `service_role` grants. This is the future home of operator watermarks (e.g. `hook_reenable_ts` consumed by `observe-rls-burn-readiness.ts` as the T0 lower bound — B-H1 mitigation). The script does NOT yet read this row dynamically; in the current cut the script falls back to `auth.users.last_sign_in_at` only. Wiring is deferred (see §4).

3. **Observability script schema fix + DR scaffolding.** `scripts/observe-rls-burn-readiness.ts` queries `audit_log` via PostgREST. The v1 of the script used the wrong column names (`action_type`, `created_at`); Cut B replaces them with the migration-correct `action` + `occurred_at`. The DR runbook (`docs/runbooks/v0.2.6-rls-burn-rollback.md`) was updated with `sa-east-1` specifics. A seed script `scripts/seed-preview-v026-w1t1-5b.ts` UPSERTs the `hook_reenable_ts` row against the preview branch only (with a `preventProdMisfire` guard against the prod `project_ref`).

Cut B-truncado **does not** ship: W1.T2 audit-log writer producers for `claim_missing`/`active_tenant_null` from the hook side, W2 RLS-burn migrations, dynamic `app_config` reads, or production deploy. Those remain in their planned waves.

## 2. Functional contracts

### 2.1 `isApprovalGateBypassed()` — kill switch helper

**File:** `src/lib/runtime-config.ts:54-56`
**Env var (constructed via Sentinel-evading bracket-join, line 28):** `APPROVAL_GATE_ENABLED`
**Reading semantics:** read **once at module load** via `readOptEnv` (line 18-21, 46). Subsequent calls return the frozen value; flipping the env var at runtime has **no effect** until the Node process restarts.

| Env var value | `isApprovalGateBypassed()` returns | Effect on guards                                                                     |
| ------------- | ---------------------------------- | ------------------------------------------------------------------------------------ |
| unset         | `false`                            | Normal enforcement.                                                                  |
| `""` (empty)  | `false` (treated as unset by L20)  | Normal enforcement.                                                                  |
| `"0"` exactly | `true`                             | Bypass engaged: both guard entrypoints return sentinel tenant + emit forensic audit. |
| any other     | `false`                            | Normal enforcement.                                                                  |

Comment at `runtime-config.ts:43-45` documents this: "Any other value (or unset) → gate enforced. Module-load read freezes the value; restart required to flip."

### 2.2 `requireActiveTenantOrRedirect` — Pages / Server Components entrypoint

**File:** `src/lib/auth/guard.ts:171-191` (NOT `requireActiveTenantPage` as the task brief named it — the file uses `requireActiveTenantOrRedirect`).
**Return shape on success:** `Promise<{ user: AuthUser; tenantId: string }>` (alias `RequireActiveTenantPageResult` at L154-157).
**On unauthenticated:** `redirect("/login")` via `requireUser()` (delegated at L172, defined L55-62).
**On no/invalid claim:** emits dedup-gated audit row via `emitTenantClaimAudit` (L127-152), then `redirect("/login?e=e07")` (L188). Reason code `e07` is opaque (does not leak the discriminator — see comment L168-170).
**On bypass (`isApprovalGateBypassed() === true`):** emits forensic `approval_gate_bypassed` audit (L178), returns `{ user, tenantId: "__bypass__" }` (sentinel from L18, L179).

### 2.3 `requireActiveTenantOrResponse` — API Route Handlers entrypoint

**File:** `src/lib/auth/guard.ts:205-241` (NOT `requireActiveTenantApi` as the brief named it).
**Return shape:** discriminated union `RequireActiveTenantApiResult` (L193-195):

- `{ ok: true, user, tenantId }` on success / bypass
- `{ ok: false, response: NextResponse }` on rejection

**Status codes:**
| Branch | Status | Body |
| ------------------- | ------ | ----------------------------------------------- |
| Unauthenticated | 401 | `{ error: "unauthorized", code: "E_AUTH" }` |
| No/invalid claim | 403 | `{ error: "forbidden", code: "E_TENANT_CLAIM" }` |
| Bypass | (n/a — returns ok:true) |

Body shape strict — test `B5` (auth-guard-tenant.test.ts:344-365) asserts only `{error, code}` and absence of `user.id`/`jti`/`email` substrings. **Never calls `redirect()`** (would crash in a Route Handler — `redirect` is server-component-only, see comment guard.ts:162-167).

### 2.4 `approval_gate_bypassed` audit action

**Writer:** `emitApprovalGateBypassAudit` (guard.ts:27-53). Distinct from `emitTenantClaimAudit` because the dedup gate (audit_dedup PK on `(jwt_jti, action)`) only fires for the `TENANT_CLAIM_ACTIONS` set (`audit.ts:70-73`: `"claim_missing"`, `"active_tenant_null"`). Bypass events bypass dedup entirely — **every bypass call writes one row.**

**Payload emitted (guard.ts:34-39):**

```
{
  action: "approval_gate_bypassed",
  jwt_jti: <extractJti from access_token, undefined if absent>,
  actor_user_id: user.id,
  metadata: { entrypoint: "page" | "api" }
}
```

**Failure handling:** `emitApprovalGateBypassAudit` **swallows audit errors** (try/catch at guard.ts:33-52, logs structured JSON to stderr but does NOT rethrow). Comment at L23-26 rationale: "bypass already implies we are in a degraded state". Contrast with `emitTenantClaimAudit` (L127-152) which **rethrows** to preserve hash-chain monotonicity.

**RLS read scope:** `audit_log` table reads via `service_role` only (W1.T2 migration `20260514_v025_005_audit_log.sql`). No authenticated/anon SELECT path.
