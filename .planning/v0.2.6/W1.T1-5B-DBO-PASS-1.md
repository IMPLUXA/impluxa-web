## 1. Index inventory (inferido)

### T0 anchor index — supports first-claim-mint window query

- **Table+cols:** `public.audit_log (action, occurred_at DESC)`
- **Type:** partial composite btree
- **WHERE:** `action IN ('claim_missing','active_tenant_null')`
- **CONCURRENTLY:** N/A — propaga a particiones via parent (ver D-H1). En cada particion: **Y** (CREATE INDEX CONCURRENTLY no permitido dentro de migrations transaccionales Supabase; debe correrse fuera de `BEGIN`. En preview/prod, ejecutar split: migration crea parent index, follow-up script ejecuta `CREATE INDEX CONCURRENTLY` por particion existente, rotate function se modifica para auto-crear).
- **Storage:** ~3 rows hoy. Post-launch tenant chico ~10/dia x 30dias = 300 rows/particion. Index ~16KB/particion negligible.
- **Query patterns served:** `observe-rls-burn-readiness.ts:189-198` post-fix (B-R1 corrige `action_type`→`action`, `created_at`→`occurred_at`). Filtra los 2 action values en ventana `> windowStart`.
- **Partitioned table:** YES. Crear como **partitioned index** (sin `ON ONLY`) para auto-propagar.

### B-R1 index — corrected observability columns

- **Table+cols:** mismo que T0 anchor. **B-R1 NO es un index separado**, es la corrección del script para usar `action`/`occurred_at`. El index que lo soporta es T0 anchor.
- **Si Senior PM intenta separarlos:** alarma — duplicación.
- **Recomendación:** unificar T0 anchor + B-R1 en un único índice físico. Renombrar conceptualmente "T0 anchor (B-R1-aligned)".

### B1 index — rotate function index propagation fix

- **NO es un index nuevo.** Es **fix estructural** al `audit_log_rotate_partitions()` y a los 3 indexes existentes del parent (`audit_log_pkey`, `audit_log_acting_tenant_occurred_idx`, `audit_log_actor_user_occurred_idx` están `ON ONLY` → particiones futuras vacías de índices).
- **Acción:**
  1. `DROP INDEX` los 3 parent indexes + recrear sin `ON ONLY` (partitioned index).
  2. Modificar `audit_log_rotate_partitions()` para NO requerir `CREATE INDEX` manual — el partitioned-index auto-attach lo hace en `CREATE TABLE ... PARTITION OF`.
- **CONCURRENTLY:** **Y obligatorio en prod** (DROP+CREATE índice del PK partitioned es DDL fuerte → ACCESS EXCLUSIVE sobre parent → bloquea writers ~segundos a minutos). **Blocker preview→prod.**

### B5 index — DR/regional outage observability

- **Table+cols:** `public.audit_log (occurred_at DESC)` global o `(actor_user_id, occurred_at DESC) WHERE action IN ('claim_missing','active_tenant_null')`.
- **Type:** partial composite btree.
- **WHERE:** mismo que T0 anchor (subset).
- **CONCURRENTLY:** Y.
- **Storage:** ~negligible single-tenant.
- **Query patterns served:** post-outage forensic — `WHERE actor_user_id = X AND action = Y ORDER BY occurred_at` para reconstrucción ventana DR (db-second-pass §B5 sesion 9a).
- **Partitioned table:** YES, sin `ON ONLY`.
- **Solapamiento con T0 anchor:** parcial (mismo WHERE, distintas cols leading). Considerar si T0 anchor con `(actor_user_id, action, occurred_at)` sirve a ambos. Ver D-H2.

## 2. Lock + downtime risk

- **B1 fix (DROP+recreate parent indexes sin ON ONLY)**: ACCESS EXCLUSIVE sobre parent + cada particion. En prod Hakuna (3 rows hoy) es sub-seg, pero en partitioned table con N particiones es serializado. **Mitigación obligatoria:** `CREATE INDEX CONCURRENTLY` sobre el partitioned table NO está soportado en Postgres ≤15 cuando es partitioned-index (PG16+ sí). Verificar versión Supabase managed.
- **T0 anchor + B5 (nuevos partitioned indexes)**: ídem — partitioned index creation no acepta CONCURRENTLY en PG<16. Path: crear índice ON ONLY parent + CONCURRENTLY en cada partición + ATTACH PARTITION INDEX manual (procedimiento documentado Supabase).
- **Interacción con `audit_log_rotate_partitions()` (8f0addf double-buffer)**: si B1 se aplica DESPUES del próximo cron tick (ya creó partición sin índices), las particiones huérfanas necesitan backfill manual. **Aplicar B1 ANTES de pg_cron habilitado** (pg_cron sigue NO instalado per db-second-pass §E3).

## 3. Pre-existing indexes overlap

Indexes actuales sobre `audit_log` parent (ON ONLY):

1. `audit_log_pkey (occurred_at, id)` — PK partitioned.
2. `audit_log_acting_tenant_occurred_idx (acting_as_tenant_id, occurred_at DESC)`.
3. `audit_log_actor_user_occurred_idx (actor_user_id, occurred_at DESC)`.

**Solapamientos con propuestos:**

- T0 anchor `(action, occurred_at DESC) WHERE action IN (...)` vs PK `(occurred_at, id)`: PK ordena por occurred_at primero — barre toda la partición para filtrar `action`. T0 anchor es ortogonal, **no redundante**, necesario.
- B5 `(actor_user_id, occurred_at DESC) WHERE action IN (...)`: **subset cubierto** por `audit_log_actor_user_occurred_idx (actor_user_id, occurred_at DESC)` general — el partial WHERE reduce footprint pero el general ya sirve. **D-H2 candidato.**

## 4. Bloat / vacuum / autovacuum

- `audit_log` es **append-only insert + hash chain** → zero updates, near-zero deletes (rotate solo crea, no dropea). Bloat mínimo. Autovacuum default OK.
- `audit_dedup` (commit 18dfdb0) tiene `autovacuum_vacuum_scale_factor = 0.05` ya tuned para alta delete-rate del cron diario — correcto.
- **TTL 7 días hardcoded** en cron job → al go-live volume (~200K rows/día per db-second-pass extrapolation), el partial index sobre `first_seen_at` (creado en migration) puede acumular dead tuples post-DELETE batch. Confirmar autovacuum lo limpia entre runs.
- Hash chain append-only → ningún índice se reordena en hot path. Bueno.
