## 2. Why "truncado" — decision rationale

### Original W1.T1 scope (pre-truncation, per PLAN.md s6)

Wave 1 was framed as 3 tasks: W1.T1 verify observability script (~10 min) + W1.T2 add audit_log writers (~45 min) + W1.T3 smoke-test e2e (~20 min). Total ~75 min active.

### What actually emerged in s13

During Two-Pass extended caso #7 (6 agents, 15 HIGH consolidated), the Squad detected:

1. **B-R1 script schema mismatch** — the observability script queried `action_type` / `created_at` columns that don't exist in the current `audit_log` schema. Script would have hard-failed on first prod telemetry pull. Required fix BEFORE anything else.
2. **B-COLD-1 APPROVAL_GATE bypass primitive** — `guard.ts` had no kill-switch path. The 5-layer mitigation stack (per SECURITY-REVIEW Veredicto 2 L4) calls out `APPROVAL_GATE_ENABLED` as app-layer escape; the primitive was missing, not just the env var. Added.
3. **B1 partition false alarm** — initially flagged HIGH; cold round verification via `pg_inherits` proved it was a representation artifact. Net savings ~2h.

### The CEO decision to truncate (s13)

Faced with: ship a minimum coherent subset NOW vs. block on full W1.T2 audit writers + their migration. The CEO chose **Cut B-truncado** = ship 5 deliverables (5B.3 / 5B.3.bis / 5B.4 / 5B.5 / 5B.6) that:

- Fix the load-bearing B-R1 script bug.
- Add the kill-switch primitive (5-layer L4).
- Stand up `app_config` skeleton (future GUC TTL parametrization vehicle, see `pre_flipeo_hakuna_live_blockers.audit_dedup_ttl_parametrizable_gdpr`).
- Land DR-region runbook section.
- Validate the whole bundle on preview branch.

W1.T2 audit_log writers got pushed to a fresh W1.T2 cycle in s15+. Reason: writers depend on hook re-enable being live in prod with telemetry actively emitting — and the Squad wanted clean separation between "infrastructure ready" (5.B) and "observability emitters live" (W1.T2).

### Net assessment

The truncation is **scope reduction, not scope creep eliminated**. The deferred items remain in v0.2.6's critical path; they just live in W1.T2 fresh instead of W1.T1. No items moved to v0.2.7+ as a consequence of 5.B.

## 4. UAT plan for Cut B feature

**Audience**: a hypothetical fresh on-call engineer (CEO equivalent) who didn't write the code. UAT proves the feature works without insider knowledge.

### UAT-1 — APPROVAL_GATE bypass behavior

**Setup**: preview branch + Vercel preview deployment.

1. Set env flag `APPROVAL_GATE_ENABLED=false` in Vercel preview (or local dotenv file).
2. Hit a protected page that goes through `guard.ts` (e.g., `/admin/<tenant>/sites`).
3. Hit an API route that goes through `guard.ts` (e.g., `GET /api/sites`).
4. Verify both succeed without claim enforcement.
5. Check `audit_log` for `event_type='approval_gate_bypassed'` warn entries — should have one per call.
6. Set flag back to `true` (or unset), redeploy, verify claim enforcement returns.

**Pass criteria**: 5/5 of (1) page bypass works, (2) API bypass works, (3) audit warn fires both entrypoints, (4) entries have valid hash-chain linkage to prior audit_log row, (5) re-enable returns enforcement.

### UAT-2 — Audit chain integrity post-bypass event

**Setup**: same preview environment as UAT-1.

1. Capture pre-test `audit_log` MAX(id) and last row hash.
2. Trigger 3 bypass events (UAT-1 steps 2-3, three times).
3. Run existing v0.2.5 hash chain integrity test: `npm test -- audit-log-hash-chain`.
4. Manually inspect: new 3 rows present, each `prev_hash` = predecessor's `hash`, no gaps.

**Pass criteria**: chain test green + manual inspection confirms no gaps.

### UAT-3 — DR runbook walk-through

**Setup**: fresh terminal, on-call engineer who has NEVER opened the file.

1. Open `docs/runbooks/v0.2.6-rls-burn-rollback.md`.
2. Read the new sa-east-1 DR section end-to-end.
3. Without running commands, the engineer should be able to answer:
   - Where is the failover DB region?
   - What's the RPO target?
   - What's the RTO target?
   - Who do I page first if primary region down?
   - Is there a tested recovery drill date?

**Pass criteria**: all 5 questions answerable from the runbook alone. If any "needs more context elsewhere" → runbook fails UAT, fix in 5B.7-5B.10.

### UAT-4 — `app_config` table migration idempotency

**Setup**: preview branch with `app_config` migration already applied.

1. Re-apply the migration: `supabase db push --linked` (against preview).
2. Verify no error, no schema drift (`supabase db diff` returns empty).
3. Insert a test row `('test_key', 'test_value')` via SQL.
4. Re-apply migration AGAIN — verify row persists (migration should not truncate).
5. Drop migration check entry and re-apply — verify clean state restoration.

**Pass criteria**: migration is idempotent + non-destructive + reversible.

### UAT-5 — Observability script schema query success

**Setup**: preview branch DSN.

1. Run `npx tsx scripts/observe-rls-burn-readiness.ts --t0 <recent_iso> --json`.
2. Verify output is parseable JSON with keys `{token_mints, claim_missing, active_tenant_null}`.
3. Verify zero schema errors (no `column "action_type" does not exist`).
4. Run with `--since-first-claim-mint` flag (OQ-4 LOCK mode) — verify it queries `auth.users.last_sign_in_at MIN` correctly.

**Pass criteria**: script returns JSON + no schema errors + both modes work.
